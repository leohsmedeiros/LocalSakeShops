# Implementation Plan: Sake Shop List & Detail

**Branch**: `002-shop-list-detail` | **Date**: 2026-06-23 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/002-shop-list-detail/spec.md`

## Summary

Build a two-screen feature (list + detail) for browsing local sake shops. Data is loaded
from the bundled `sake_shops.json` using a generic, re-usable `BundledJSONDataSource`.
All layers follow Clean Architecture (Domain → Data → Presentation). ViewModels manage
state via a single `ViewState<T>` enum. Navigation uses `NavigationStack` with typed
destinations. All UI consumes the existing design system tokens. The feature ships with
complete unit tests (Swift Testing) and UI snapshot tests (XCTAttachment).

## Technical Context

**Language/Version**: Swift 5.9 / Swift 5.10 (Xcode 15+)

**Primary Dependencies**: SwiftUI (Apple-native). `AsyncImage` for remote image loading.
No third-party libraries.

**Storage**: Bundled JSON file (`sake_shops.json`) — read-only, no persistence layer
needed for this feature.

**Testing**: Swift Testing (`@Suite`, `@Test`, `#expect`) for unit tests; `XCUITest` +
`XCTAttachment` for UI/snapshot tests.

**Target Platform**: iOS 16+

**Project Type**: iOS mobile app — feature-first modular architecture within a single
Xcode target. `PBXFileSystemSynchronizedRootGroup` auto-includes new source files.

**Performance Goals**: List visible within 1 second of launch (JSON is tiny: 10 shops).
60fps scrolling baseline. Remote images load asynchronously via `AsyncImage` — list
render is not blocked by image loads.

**Constraints**: No third-party libraries. All UI tokens from `docs/DESIGN.md`.
`AsyncImage` requires an internet connection for shop images; a placeholder is shown
when the URL is null or the network is unavailable.

**Scale/Scope**: 10 shops in bundled JSON. ~15 source files, ~8 test files. Single
feature, two screens.

**Structure Decision**: Feature-first modular layout within a single Xcode target.
`Domain/` and `Data/` are shared infrastructure for this feature and will grow naturally

that satisfies the explicit re-usability acceptance criterion.

## Reusability Design

The re-usability requirement is satisfied at two layers:

1. **`BundledJSONDataSource`** — generic on `T: Decodable`. Swapping to another JSON
   file requires only a new DTO, entity, mapper, and repository — zero changes to the
   data source itself.

2. **Protocol-driven layers** — `BundledJSONDataSourceProtocol` and
   `SakeShopRepositoryProtocol` mean a remote API implementation can replace the bundled
   JSON source with no changes to the use case or ViewModel layers.
